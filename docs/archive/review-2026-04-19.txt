2. 评价

整体评价：Stage 12 是一个明显进步版，但不是最终稳定版。

它已经修掉了 Stage 11 里最关键的大部分问题：

第一，tests_run: 1 硬编码问题基本修掉了。现在 after_edit_update.py 里有 parse_test_count()，并且能解析 unittest、pytest、node:test、vitest、jest 的常见输出。无法解析时会返回：

{
  "tests_run": null,
  "test_count_status": "unknown"
}

这比之前诚实很多。

第二，dependency fingerprint 的正常路径已经接上了。incremental_refresh.py 现在会追踪：

package.json
package-lock.json
pnpm-lock.yaml
yarn.lock
bun.lock
bun.lockb
pyproject.toml
requirements.txt
requirements-dev.txt
poetry.lock
Pipfile
Pipfile.lock
tsconfig.json
vite.config.*
next.config.*

trust_policy.py 也已经区分：

not_applicable
unknown
unchanged
changed

第三，TS/JS parser 的几个危险点有明显加强。现在它不是简单遇到第一个 { 就认为是函数 body，而是加了 find_js_body_open_brace()，并覆盖了 TypeScript 参数类型、返回类型、template、regex 等边界。Stage 11 的 TS/JS 回归测试能过，说明这个方向是有效的。

第四，CONTEXT_MISSING 现在不只是自然语言提示，而是有 recovery_commands。这对弱模型非常重要。

第五，health 命令已经出现了。这是很好的产品方向，因为它让 agent 不必自己猜当前仓库状态。

第六，Stage 12 对 self-hosting 的问题开始收口。.code-impact-guardian/config.json 现在明确针对 guardian repo 本身，project_root 是 "."，source globs 也覆盖了：

scripts/*.py
scripts/**/*.py
.agents/skills/code-impact-guardian/*.py
.agents/skills/code-impact-guardian/**/*.py

同时测试确认 nested dist/ copy 不会被匹配进来。

但是现在还存在几个问题。

问题 A：Stage 12 review bundle 不完整

STAGE12_REVIEW_GUIDE.md 说这是 expanded review bundle，并且包含 full tests workflow suite。可是测试依赖的 benchmark/ 目录没有被包含。

这会造成两个后果：

一是外部 reviewer 无法验证 Stage 9 / Stage 10 的完整回归。

二是 Codex / agent 会一直跑测试、失败、重试，但失败原因不是实现逻辑，而是 fixture 缺失。这会浪费大量开发期 token 和时间。

这个必须先修。

问题 B：--full-rebuild 会绕过 dependency trust 降级

这是我检查时发现的隐藏问题，比测试失败更关键。

我手动构造了一个带 requirements.txt 的最小项目，然后跑：

python .agents/skills/code-impact-guardian/cig.py build --workspace-root <repo> --config <config> --full-rebuild

在第一次构建、没有 previous manifest 的情况下，输出是：

dependency_fingerprint_status = unknown
graph_trust = high
reason_codes = ["NO_PREVIOUS_MANIFEST", "DEPENDENCY_FINGERPRINT_UNKNOWN", "FORCED_FULL_REBUILD"]

我又测试了依赖文件变更后强制 full rebuild 的情况，输出是：

dependency_fingerprint_status = changed
graph_trust = high
reason_codes = ["DEPENDENCY_FINGERPRINT_CHANGED", "FORCED_FULL_REBUILD"]

这违反了 Stage 11 的核心规则：

dependency_fingerprint_status = unknown 时，不得 high trust
dependency_fingerprint_status = changed 时，不得 high trust

问题位置在：

.agents/skills/code-impact-guardian/scripts/build_graph.py

当前逻辑类似：

if force_full:
    decision["build_mode"] = "full"
    decision["trust_level"] = "high"
    decision["graph_trust"] = "high"
    decision["reason_codes"] = [*decision.get("reason_codes", []), "FORCED_FULL_REBUILD"]

这会把前面 trust_policy.py 里正确做出的降级判断覆盖掉。

这是 P0，需要修。

问题 C：测试成本仍然偏重

Stage 11 测试 15 个用例跑了约 169 秒。不是灾难，但对开发期来说确实重。

原因不是单个 parser 测试慢，而是每个测试都会复制 skill、setup、build、跑 CLI，有大量端到端成本。

这对“开发这个 skill 本体”是可接受但偏重的；对“普通用户真实使用”不是同一个概念。普通用户不会跑 test_stage11_workflow.py，但真实使用时 finish 仍会跑配置里的 test command，而且当前默认还是偏 full suite。

所以测试风暴问题还没有从产品层彻底解决。

问题 D：iter_matching_files() 结果上能避开 dist，但遍历上仍会扫整个树

现在 iter_matching_files() 是：

for path in project_root.rglob("*"):
    ...

然后再用 pattern 过滤。

Stage 12 的测试确认了 nested dist/ 里的文件不会被匹配，但它没有解决性能问题：rglob("*") 仍然会进入 dist/、node_modules/、.next/、coverage/ 等大型目录。

这不是 correctness P0，但对真实 React / Electron / Obsidian / Tauri 项目会影响很大。下一阶段应该改成可剪枝的 os.walk()，在遍历层就跳过这些目录。

P0-2：修复 --full-rebuild 的 trust 覆盖

修改文件：

.agents/skills/code-impact-guardian/scripts/build_graph.py

当前问题是 force full rebuild 时直接写死：

decision["trust_level"] = "high"
decision["graph_trust"] = "high"

建议改成：

if force_full:
    decision["build_mode"] = "full"
    decision["execution_mode"] = "full"
    decision["reason_codes"] = [*decision.get("reason_codes", []), "FORCED_FULL_REBUILD"]

    risk_codes = set(decision.get("reason_codes", []))
    dependency_status = decision.get("dependency_fingerprint_status")

    if dependency_status in {"changed", "unknown"}:
        decision["trust_level"] = "medium"
        decision["graph_trust"] = "medium"
    elif "NO_PREVIOUS_MANIFEST" in risk_codes:
        decision["trust_level"] = "medium"
        decision["graph_trust"] = "medium"
    elif "GENERATED_OR_CACHE_NOISE_PRESENT" in risk_codes:
        decision["trust_level"] = "low"
        decision["graph_trust"] = "low"
    else:
        decision["trust_level"] = decision.get("trust_level", "high")
        decision["graph_trust"] = decision.get("graph_trust", decision["trust_level"])

核心原则是：

full rebuild 可以让 graph fresh
但不能自动抹掉 dependency unknown / changed 的风险
P0-3：新增两个回归测试

加到：

tests/test_stage11_workflow.py

或者新建：

tests/test_stage12_workflow.py

建议新增：

test_full_rebuild_does_not_raise_dependency_unknown_to_high
test_full_rebuild_does_not_raise_dependency_changed_to_high

验收条件：

self.assertEqual(decision["dependency_fingerprint_status"], "unknown")

以及：

self.assertEqual(decision["dependency_fingerprint_status"], "changed")
self.assertNotEqual(decision["graph_trust"], "high")

这两个测试必须加，否则以后很容易再次被覆盖。
