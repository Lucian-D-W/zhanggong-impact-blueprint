项目背景：这个项目的开发背景，可以概括成一句话：你想做的不是一个“会写代码”的 agent，而是一套给 agent 加护栏的通用 Skill。它的目标，不是替代大模型编程，而是在 vibe coding 过程中，减少“改了 A 却悄悄破坏 B”的情况，让 AI 在动手前先看影响范围、动手后留下结构化记录、失败时也能给出下一步恢复建议。 最初的动机，来自一个很真实的问题：大模型在协作写代码时，常常只盯住眼前这条链路，能把当前问题修掉，但对更远的调用关系、规则约束、测试覆盖和上下游影响，不一定会主动看见。于是你想做一个能被反复复用的 Skill：把它放进项目里，AI 每次改代码前都先跑一遍分析，生成 impact report，再允许编辑，改完后再更新图谱、测试结果和日志。这样，AI 不再只是“凭感觉改”，而是“带着施工检查单改”。 这个项目的核心能力，围绕四类节点和五类边展开。节点包括 file、function、test、rule；边包括 DEFINES、CALLS、IMPORTS、COVERS、GOVERNS。它会把这些结构事实落进 SQLite 里，再配合 report、status、handoff、错误日志，形成一套“改前体检—改后复查—失败可恢复”的链路管理流程。你可以把它理解成：Git 负责记住“以前改了什么”，而这个 Skill 负责回答“现在改这里，会波及谁”。技术主管指出的漏洞也正说明了这套系统为什么要继续打磨：比如 Python 类方法还是盲区、TS/JS 用大括号计数判断函数边界会被字符串和模板字面量欺骗、stale 图谱有时被错误地标成 high trust、recent task 可能把 seed 排名带偏、测试通过又容易给用户造成“安全了”的错觉。 从产品角度看，这个项目要解决的并不只是“有没有图谱”，而是“普通用户和普通 agent 感不感觉它有价值”。这也是为什么后来又补了 setup、doctor、detect、build、analyze、finish、status、handoff 等入口，希望把它做成一个复制进项目就能用的 skill，而不是只适合开发者自己调试的框架。产品同事提出的重点也很准确：Skill 的触发语言要更强约束、analyze 之后要有一两句自然语言向用户解释“这次影响了什么”、第一次缺配置时要自动 setup、SKILL 文档本体要更瘦、传播上最好有一个一眼能看懂的演示。 就技术栈和使用场景来说，这个项目已经不是单语言玩具了。你现实里最核心的是 React + TypeScript，桌面端会碰 Electron、Tauri，插件方向会碰 Obsidian，自动化和抓取会用 Node.js + Playwright + SQLite，也会用到 Python；而数据库和后端逻辑里，PostgreSQL / 通用 SQL 的价值也越来越高。所以这个项目后续不是单纯“再加语言”，而是要继续往两个方向推进：一是通用性，也就是 Python + TS/JS + React + PostgreSQL/通用 SQL 这些主链路要变得足够稳定；二是能用性，也就是默认 brief 更短更稳、seed 选择更聪明、自动从 git diff / patch / changed-file 推断上下文、增量刷新和全量重建的信任边界更清楚、错误恢复更具体，最好让普通 agent 能自然地顺着 Skill 走，而不是每次都要你在旁边解释。 所以，这个项目的本质，不是“做一个更会写代码的 AI”，而是“做一个让 AI 在真实开发中更少断线、更少误伤、更可交接、更可持续升级的通用 Skill”。这也是它为什么值得继续做下去：它瞄准的不是某一门语言的 parser，而是 AI 编程协作里长期存在的链路管理问题。
